## 三、降级链改进建议

### 3.1 当前问题

| 问题 | 表现 | 根因 |
|------|------|------|
| 429雪崩 | 多次429后所有模型都429 | 限流是账号级，切换模型没用 |
| 降级链是线性的 | 一个挂了遍历全部超时 | 没有提前检测可用性 |
| 没有冷却机制 | 刚429立刻切下一个，也可能429 | 同一provider共用限额 |

### 3.2 改进方案

```
改进后降级链（代码任务）：

1. SiliconFlow DS4 Pro (首选)
   ↓ 429/超时 → 冷却标记（5分钟内不重试）
2. 智谱 GLM-4.7 (兜底，跨provider）
   ↓ 429/超时 → 上报 CC 人工处理
3. 等待人工指令
```

核心改进点：

**A. Provider级冷却（最关键）**
```js
const providerCoolDown = new Map();
// 429时标记该provider冷却5分钟
providerCoolDown.set('siliconflow', Date.now() + 5 * 60 * 1000);
// 下次选择模型时跳过冷却中的provider
```

**B. 跨provider优先**
- SiliconFlow 429 → 切智谱（不同API Key、不同账户）
- 智谱 429 → 切 TaoToken（第三个provider）
- 同一 provider 内切模型没意义——429是账户级

**C. 自适应超时**
- 日常任务：30s 超时（Flash速度快）
- 代码任务：120s 超时（Pro推理时间长）
- 文件读写：15s 超时（纯IO不应耗时）
