## 1. 决策背景

技术方案 v1.1 选定 Preact 10 + Vite 5 作为前端框架。CC 在复盘意见中提出 Vanilla JS 在后续迭代时维护成本会指数上升，建议引入轻量框架。小马要求三条路线（Vanilla JS / Preact / Alpine.js）做正式对比，产出明确推荐。

本文档从团队工作室的三个核心场景出发，给出适用性分析和推荐结论。

## 2. 候选方案概览

| 维度 | Vanilla JS | Preact 10 | Alpine.js 3 |
|------|-----------|-----------|-------------|
| 体积 | 0 KB（无依赖） | ~4 KB gzip | ~15 KB gzip |
| 模式 | 命令式 DOM 操作 | 声明式 JSX 组件 | 声明式 HTML 指令 |
| 状态管理 | 手写（变量 + 手动 DOM 更新） | Signals / hooks / class | `x-data` 响应式对象 |
| 构建工具 | 不需要（或 Vite 仅做 HMR） | Vite + @preact/preset-vite | Vite（可选，也可以纯 CDN） |
| 学习成本 | 无 | 低（React 语法子集） | 极低（HTML 属性驱动） |
| 生态 | 无 | React 生态兼容 | 独立小生态 |
| SSR | 不适用 | 支持 | 支持 |
| 适合场景 | 极简页面、无交互 | 组件化 SPA、中等复杂度 | 渐进增强、服务端渲染页面 |

## 3. 三个核心场景分析

### 3.1 场景一：角色像素动画

**需求**：Canvas 2D 逐格渲染像素网格，CSS 控制动画状态切换（idle / working / talking / walking / error / offline），角色行走移动插值。

**分析**：

| 方案 | 适用性 | 说明 |
|------|--------|------|
| Vanilla JS | **完全适配** | Canvas 渲染本身是命令式的，requestAnimationFrame 循环 + 手动 ctx.fillRect，无需框架介入。动画状态机用 switch-case 即可。 |
| Preact | **中性** | `useEffect` 包裹 game loop，`useRef` 持有 canvas 引用，Signals 驱动状态更新触发重绘。框架层不直接参与渲染，但提供干净的生命周期管理。 |
| Alpine.js | **不适用** | Alpine.js 设计目标是增强 HTML，不擅长 Canvas 渲染循环。用 `x-data` 管理角色状态可以，但 game loop 仍需手写，框架层无收益。 |

**结论**：角色动画层是纯 Canvas 命令式代码，三种方案差异极小。Vanilla JS 零开销，Preact 提供生命周期管理，Alpine.js 在此场景无优势。

### 3.2 场景二：实时聊天面板

**需求**：三方群聊（KK / CC / 小马），消息列表动态渲染，KK 靠右金色、CC/小马 靠左，支持 Markdown，系统消息（上下线通知），滚动到底部。

**分析**：

| 方案 | 适用性 | 说明 |
|------|--------|------|
| Vanilla JS | **痛苦** | 每收到一条 WebSocket 消息，需要：createElement → 设置 class/内容/头像 → appendChild → 滚动到底部。消息列表更新需要手动管理 DOM 节点增删。Markdown 渲染后 innerHTML 赋值需手动 XSS 过滤。100+ 条消息后 DOM 操作代码量线性增长。 |
| Preact | **天然适配** | `messages.map()` 声明式渲染，JSX 天然支持条件渲染（KK 靠右/CC 靠左），组件化拆分（MessageItem / ChatInput / OfflineIndicator），Signals 驱动自动更新。 |
| Alpine.js | **可用但笨重** | `x-for` 循环渲染消息列表，`x-if` 条件判断对齐方式。但 Alpine.js 的模板写在 HTML 属性里，复杂嵌套（消息气泡 + 头像 + 时间戳 + Markdown）会让 HTML 变得臃肿难读。 |

**代码量对比（消息列表渲染，估算）**：

| 方案 | 代码行数 | 说明 |
|------|---------|------|
| Vanilla JS | ~80-120 行 | 手动 DOM 构建 + 事件绑定 + 滚动控制 |
| Preact | ~30-40 行 | JSX 声明式组件 |
| Alpine.js | ~50-60 行 | HTML 模板 + x-for/x-if 指令 |

### 3.3 场景三：状态同步

**需求**：WebSocket 接收状态变更事件 → 更新 Agent 状态（status / activity / progress / location）→ 联动 Canvas 角色动画 + 聊天面板在线状态 + 底部状态栏。多处 UI 共享同一份状态。

**分析**：

| 方案 | 适用性 | 说明 |
|------|--------|------|
| Vanilla JS | **最痛苦** | 状态更新后需要手动通知所有订阅者：Canvas 重绘、聊天面板在线指示灯、状态栏文字。手写发布-订阅模式或全局变量 + 轮询检查。代码量大且容易遗漏更新。 |
| Preact + Signals | **最优解** | `signal()` 定义状态 → 任何组件读取 signal 自动订阅 → WebSocket 事件更新 signal → 所有依赖组件自动重绘。Canvas 层通过 `effect()` 监听信号变化触发重绘。零样板代码。 |
| Alpine.js | **可用** | `Alpine.store()` 定义全局状态，`$store` 在模板中绑定。但 Alpine.js 的 store 不是细粒度响应式的——store 里任何属性变化会触发所有绑定该 store 的元素更新，无法精确到单个 Agent 的单个字段。3 个 Agent × 5 个字段 = 15 个状态项，Alpine.js 的粗粒度更新会导致不必要的 DOM diff。 |

**状态同步架构对比**：

```
Vanilla JS:
  WS.onmessage → switch(type) → 手动更新变量 → 手动调用 renderCanvas() / appendMessage() / updateStatusBar()
  问题：新增一个 UI 组件就要加一个手动调用点

Preact + Signals:
  WS.onmessage → handleWSEvent() → agents.value = {...} → 自动触发所有依赖组件重绘
  优势：新增组件直接读 signal，无需修改事件处理代码

Alpine.js:
  WS.onmessage → switch(type) → Alpine.store('agents').cc.status = 'working' → 粗粒度重绘
  问题：store 粒度不够细，可能触发无关元素更新
```

## 4. 综合评分

| 维度 | 权重 | Vanilla JS | Preact | Alpine.js |
|------|:----:|:----------:|:------:|:---------:|
| 角色动画（Canvas） | 20% | 10 | 8 | 6 |
| 实时聊天（DOM 渲染） | 35% | 3 | 10 | 6 |
| 状态同步（响应式） | 30% | 2 | 10 | 6 |
| 学习成本 | 5% | 10 | 8 | 9 |
| 体积/性能 | 5% | 10 | 9 | 8 |
| 生态/可扩展性 | 5% | 2 | 9 | 5 |
| **加权总分** | 100% | **4.65** | **9.45** | **6.15** |

评分说明：10 分 = 最优，1 分 = 最差。权重基于团队工作室场景的实际代码占比分配——聊天和状态同步是主要代码量所在。

## 5. 推荐结论

### **推荐：Preact 10 + Vite 5 + @preact/signals**

理由：

1. **聊天面板是代码量最大的模块**——三方群聊包含消息列表、条件渲染、Markdown 解析、滚动控制、输入框交互。Vanilla JS 在这个模块的代码量是 Preact 的 3 倍，且维护成本随消息功能扩展线性增长（消息撤回、编辑、回复、已读等后续功能都会直接增加 DOM 操作代码）。

2. **状态同步是架构关键**——WebSocket 状态变更需要同时驱动 Canvas 动画、聊天面板、状态栏三处 UI。Preact Signals 的细粒度响应式是这个场景的最优解，Vanilla JS 需要手写 pub/sub，Alpine.js 粒度不够。

3. **迁移成本可控**——Preact API 兼容 React，团队如果后续扩展或换人，React 开发者可直接上手。Alpine.js 的 HTML 指令模式在复杂组件场景下可读性差，不利于代码审查。

4. **对现有原型的影响小**——当前原型 `原型_像素工作室.html` 是纯 HTML/CSS/JS。迁移到 Preact 的改造范围：
   - HTML 结构 → 拆成 JSX 组件（~2 天工作量）
   - CSS 动画 → 保留不动，通过 className 切换
   - Canvas 渲染逻辑 → 保留不动，包装进 useEffect
   - 像素网格数据 → 保留不动，import 进组件

### 不选 Vanilla JS 的核心原因

不是 MVP 跑不动——MVP 用 Vanilla JS 完全能做。问题是**团队工作室不是一个一次性项目**。PRD 里明确列了 P2 功能（Agent 接入、离线处理），项目优化清单里还有消息撤回、文件发送等后续需求。Vanilla JS 的维护成本在第 3-4 个功能加入后会显著上升，而这个上升是不可逆的——到时候再迁移到框架，改动量比现在直接用框架更大。

### 不选 Alpine.js 的核心原因

Alpine.js 的设计哲学是"渐进增强静态 HTML"——给已有的服务端渲染页面加交互。团队工作室是一个纯前端 SPA，没有服务端渲染的 HTML 需要增强。Alpine.js 的 `x-for` / `x-if` 在复杂嵌套组件（消息气泡 + 头像 + 时间戳 + Markdown 渲染）场景下，HTML 模板的可读性和可维护性不如 JSX。另外 Alpine.js 的 store 响应式粒度是对象级别，不是字段级别，对状态同步场景不够精确。

## 6. 如果 KK 选 Preact：对现有原型的改造成本评估

### 6.1 原型现状

`原型_像素工作室.html` 是一个单文件 HTML，包含：
- CSS 像素网格角色定义（Clawd 螃蟹 + 像素小马）
- CSS 动画（idle / working / talking / walking / error / offline）
- Canvas 2D 渲染逻辑
- 聊天面板 HTML 结构
- 简单的 JS 交互

### 6.2 改造清单

| 改造项 | 工作量 | 说明 |
|--------|--------|------|
| 项目初始化（Vite + Preact） | 0.5h | `npm create vite` + 安装 preact |
| HTML 结构 → JSX 组件拆分 | 4h | 拆成 App / PixelOffice / ChatPanel / StatusBar 等组件 |
| CSS 动画迁移 | 1h | CSS 原样复制，className 绑定到 Preact 组件 |
| Canvas 渲染 → useEffect 包装 | 2h | game loop 逻辑不变，套进 useEffect + useRef |
| 像素网格数据 → JS 模块 | 0.5h | 从 HTML 内联提取为独立 .js 文件 import |
| 状态管理 → Preact Signals | 3h | 用 signal 替换全局变量，WebSocket 事件驱动 signal 更新 |
| WebSocket 客户端 → Preact 集成 | 2h | WSClient 类保留，onmessage 回调更新 signal |
| **总计** | **~13h** | 约 1.5 个工作日 |

### 6.3 不需要改的部分

- 像素网格二维数组数据（原样 import）
- CSS 动画 keyframes 和 class（原样复制）
- Canvas 2D 渲染函数（renderCharacter 等原样复用）
- 角色状态机逻辑（原样复用）
- 聊天消息样式（原样复用）
