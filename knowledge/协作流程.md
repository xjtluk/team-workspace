# 团队协作流程

> CC（研发部Leader）与 CX（代码工程师）之间的协作规范
> 最后更新：2026-06-05

---

## 一、CC-CX 分工铁律

**CC 不动手写实现代码，只做三件事：设计方案、派发任务、审查终审。**

| 任务类型 | 谁做 | 示例 |
|----------|------|------|
| 架构设计/技术决策 | CC | 选技术栈、定API结构、设计降级链 |
| 代码实现/配置更新/批量测试 | CX | 写代码、改yaml、curl测试端点 |
| 代码审查终审 | CC | 验证CX产出，合并决策 |

**例外规则：** 1分钟内能搞定的事（配置修改、小bug修复、文件复制等），CC可直接做，不派发。

### CC-CX 互助铁律

CC 和 CX 互相负责技术问题，不找小马。

| 场景 | 正确做法 | 错误做法 |
|------|----------|----------|
| CX 挂了 | CC 直接修复 | 上报小马协调 |
| CC 挂了 | CX 直接修复 | 上报小马协调 |
| 两个都挂了 | 找小马协调 | — |
| API Key/环境问题 | 自己排查修复 | 等别人解决 |

## 二、多子任务协作协议

**适用场景**：一个任务拆成多个子任务（如：前端改UI + 后端加心跳 + watchdog重启）

### 流程

**第一步：首次分配**
CC 拆解总任务为若干子任务，分成两份：CC 做哪些、CX 做哪些，群内通知。双方同时开始。

**第二步：CX 进度同步**
CX 每完成一个子任务，在群内同步：
```
@CC [子任务完成] 子任务名称 | 改动文件 | 剩余: X个（列出剩余任务）
```

**第三步：CC 完成后主动协调**
CC 先做完所有自己的子任务。如果空闲超过 5 分钟：
1. 查看 CX 最新同步的剩余任务清单
2. 判断哪些 CX 剩余任务 CC 也能做
3. 群内通知新的任务分配：
   ```
   @CX [任务调整] 我来接手: 任务X、任务Y | 你继续做: 任务Z
   ```
4. CC 继续做新领的任务

**第四步：循环**
CX 完成当前任务后，看到 CC 更新了任务清单，按剩余任务继续做。重复第二、三步，直到所有任务完成。

**第五步：终审**
所有子任务完成后，CC 统一审查终审，确认无误后通知小马。

### 阻塞标记规则（⛔）

| 场景 | 操作 | 效果 |
|------|------|------|
| CX 卡住（技术问题/配置缺失/不确定） | 同步消息带 ⛔ | CC 立即介入，不等5分钟 |
| CX 连续2次同步无进展 | CC 主动查看 | CC 立即介入接手或指导 |
| CX 明确请求帮助 | 群内 @CC 求助 | CC 立即响应 |

## 三、TOK 自检机制

每次产出必须附带 TOK 自检，格式附在 `[完成]` 消息后面：

```
@角色 [完成] 描述 | 文件路径 | T:match O:compliant K:valid
```

| 维度 | 含义 | 检查项 |
|------|------|--------|
| **T (Task Match)** | 产出是否匹配任务需求 | 功能完整？边界覆盖？需求无遗漏？ |
| **O (Operational Compliance)** | 路径、目录、流程是否正确 | 文件落盘位置？格式规范？命名约定？ |
| **K (Key Result Valid)** | 接收方能否直接使用 | 可运行？可审查？无阻塞依赖？ |

## 四、Karpathy 编码四原则

来源：Andrej Karpathy 方法论，2026-06-01 KK 引入。

| 原则 | 含义 | 执行标准 |
|------|------|----------|
| 思考先于编码 | 不确定的先问，别瞎猜 | 收到模糊需求时先澄清，不自行脑补参数、路径、方案 |
| 简洁优先 | 代码能短就别写长 | 禁止过度工程化，不预支未来需求，不写"以后可能用到"的代码 |
| 手术式修改 | 没让你改的地方别碰 | 只修改与当前任务直接相关的文件和代码块，不顺手重构、不扩面 |
| 目标驱动 | 每一步清楚在解决什么 | 执行前自问「这一步解决了什么问题」，禁止盲跑、禁止惯性操作 |

## 五、消息协议

| 消息类型 | 格式 | 示例 |
|----------|------|------|
| 任务派发 | `@角色 [任务] 描述` | `@CX [任务] 更新API配置` |
| 代码任务派发 | `@角色 [代码] 描述 \| 参考文件 \| priority:高/中/低` | `@CX [代码] 心跳追踪 \| tasks/task.md \| priority:高` |
| 阶段完成 | `@角色 [完成] 描述 \| 文件路径` | `@CC [完成] 配置更新 \| D:/config.yaml` |
| 子任务完成 | `@CC [子任务完成] 名称 \| 改动文件 \| 剩余: X个` | `@CC [子任务完成] 看板中文 \| src/xxx.jsx \| 剩余: 2个` |
| 问题上报 | `@角色 [问题] 描述` | `@CC [问题] API认证失败` |
| TOK自检 | 附在`[完成]`消息后 | `T:match O:compliant K:valid` |

## 六、反思沉淀流程

每次犯错/被纠正/任务失败后执行：

1. `回顾日志/` 记录事实
2. `知识库/` 创建专题文档（新问题）
3. `团队守则.md` 更新规则（涉及流程时）
4. `memory/` 写入记忆条目
5. 群聊通报

---

*本流程由CC和CX共同维护，根据实践经验持续更新。*
