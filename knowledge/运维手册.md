## 一、服务管理

### 1.1 启动方式

**方式 A：Watchdog 守护进程（推荐）**

```bash
cd D:/BKS/projects/team-workspace
node scripts/watchdog.mjs        # 启动所有服务（server + cc + cx）
node scripts/watchdog.mjs --no-cx  # 不启动 CX（仅 server + cc）
```

Watchdog 自动管理：
- workspace-server（端口 3210）
- cc-listener（CC Agent 监听进程）
- cx-listener（CX Agent 监听进程）
- 崩溃自动重启（指数退避：10s → 20s → 40s → ... → 5min）
- 健康检查 / Agent 心跳监控（连续 2 次异常自动重启）
- 每日数据库备份

**方式 B：PM2（备选）**

```bash
npm run pm2:start     # pm2 start ecosystem.config.cjs
npm run pm2:status    # 查看状态
npm run pm2:logs      # 查看日志
npm run pm2:stop      # 停止所有
```

**方式 C：开发模式**

```bash
npm run dev           # 前端热更新 + 后端自动重启
```

### 1.2 停止

```bash
# Watchdog — Ctrl+C 发送 SIGINT，5秒内优雅退出
# PM2
npm run pm2:stop
```

### 1.3 重启

```bash
# 单独重启 workspace-server
npx pm2 restart workspace-server

# 单独重启 cc-listener
npx pm2 restart cc-listener

# 单独重启 cx-listener
npx pm2 restart cx-listener

# 全部重启
npm run pm2:restart
```

### 4.2 Agent 上下线

**AI Agent 自动管理：**
- Watchdog 每 60s 检查 `/api/health` 中各 Agent 的 `healthy` 状态
- 连续 2 次检测到 `healthy=false` → 自动重启对应 listener 进程

**手动干预：**
```bash
# 手动重启 CC
npx pm2 restart cc-listener

# 手动重启 CX
npx pm2 restart cx-listener
```

### 4.3 Agent 状态上报流程

```
Agent 执行任务
  → reportStatus('working', '正在编写代码', 30, 'desk')
  → POST /api/status
  → Server 更新 agents 表 + 广播 status_change
  → 前端像素面板更新角色动画和气泡

Agent 完成任务
  → reportStatus('idle', '', 0, 'sofa')
  → sendMessage('✅ 任务完成：XXX')
  → POST /api/message (仅开始/完成/异常时发消息)
  → 群聊面板显示完成消息
```

**关键约定：** 任务执行中只更新状态气泡，不发送群聊消息。

### 4.4 离线消息处理

```
Agent 离线 → 发给它的消息写入 offline_queue
Agent 上线 → POST /api/heartbeat → Server 返回 pendingMessages > 0
Agent → GET /api/history 拉取离线消息
Server → 标记 offline_queue 为 delivered
```

## 六、故障处理

### 6.1 常见问题

| 问题 | 原因 | 处理 |
|------|------|------|
| 端口 3210 被占用 | 已有进程在运行 | `netstat -ano | findstr :3210` 查 PID 后 kill |
| 数据库损坏 | 异常断电/进程崩溃 | `node scripts/backup-db.mjs --restore` 恢复 |
| Agent 心跳中断 | Listener 进程卡死 | Watchdog 自动重启，或手动 `npx pm2 restart` |
| WebSocket 断连 | 网络抖动 | 客户端自动重连（指数退避 1s→30s） |
| 前端白屏 | 构建产物损坏 | `npm run build` 重新构建 |
| API 返回 500 | 服务端异常 | 查看 `logs/workspace-server.log` 定位错误 |

### 6.2 快速恢复流程

```
1. 检查服务状态
   → curl http://localhost:3210/api/health
   → 查看 agents 字段中各成员状态

2. 如果 server 挂了
   → npm run pm2:restart
   → 等待 10s 后重新检查 health

3. 如果 Agent 挂了
   → npx pm2 restart cc-listener
   → npx pm2 restart cx-listener

4. 如果前端挂了
   → npm run build
   → 重新加载浏览器 (http://localhost:3210)

5. 如果数据库挂了
   → node scripts/backup-db.mjs --restore
   → npm run pm2:restart
```

### 6.3 数据修复

```bash
# 从备份恢复
node scripts/backup-db.mjs --restore
# 选择最近的备份时间戳恢复

# 手动清理 (通过 sql.js CLI)
npx sql.js -d data/workspace.db "DELETE FROM messages WHERE created_at < 1717171717171;"
npx sql.js -d data/workspace.db "VACUUM;"  # 回收空间
```

## 八、网络代理规范

### 代理配置

所有 Agent 进程已内置代理环境变量：
```
http_proxy=http://127.0.0.1:7897
https_proxy=http://127.0.0.1:7897
```

### 手动命令代理

```bash
# 境外站点
curl -s --proxy http://127.0.0.1:7897 https://api.github.com/xxx

# Git 临时代理
git config http.proxy http://127.0.0.1:7897
git push origin master
git config --unset http.proxy
```

### 禁止事项

| 操作 | 风险 |
|------|------|
| 修改注册表 ProxyEnable/ProxyServer | 破坏系统网络 |
| 设置系统级 HTTP_PROXY | 影响国内直连服务 |
| `git config --global http.proxy` | 永不过期，忘记清理出问题 |

国内站点（百度、阿里云、国内 API 等）**直连**，不走代理。Clash Verge 规则模式自动路由。
